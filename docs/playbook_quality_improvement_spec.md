> **版本**: v1.0 (Draft)
> **日期**: 2026-04-23

---

## 1. 背景与问题

### 1.1 当前状态

我们基于 5732 个 playbook 对 Qwen2.5-7B 进行了 GRPO 强化学习训练（213 steps / 3 epochs）。Playbook 由 LLM（豆包 Seed 2.0 Pro）从真实客服会话自动生成。

### 1.2 全量审计结果（5732 个 playbook vs 原始会话）

| 指标 | 数值 | 评估 |
|------|------|------|
| 技能映射准确率 | **47%** | ❌ 不到一半的"正确答案"和真实客服行为对得上 |
| 对话覆盖率 | 61% | ⚠️ 近四成的真实对话轮次在 playbook 中丢失 |
| 路径长度不匹配率 | **46.9%** | ❌ 近一半 playbook 比真实对话显著短 |
| 不可映射动作率 | **21%** | ⚠️ 真实客服行为中 1/5 无法用 31 个 skill 表达 |
| 买家问题捕获率 | 99% | ✅ |
| 场景-技能错配率 | 17.6% | ⚠️ |
| 有内容问题的 playbook | **78.2%** | ❌ |

### 1.3 核心矛盾

**模型在学什么？**

当前 RL 训练的本质是：给模型一段买家消息，让它从 available_skills 中选一个 skill，环境根据 transitions 判定对错。

问题在于：
1. **"正确答案"本身不准确** — 47% 的技能映射准确率意味着 playbook 标注的 golden path 有一半是 LLM 猜错的
2. **动作空间不完整** — 21% 的真实客服行为无法映射到 31 个 skill（发链接、备注订单、发图片等）
3. **对话被截断** — 46.9% 的 playbook 丢失了真实对话的后半段

**模型学完能干什么？**

以当前质量，模型能学到：
- ✅ 对话格式（`<think>...<action>...`）
- ✅ 粗粒度场景路由（售前/物流/售后的大致流程）
- ⚠️ 具体 skill 选择（准确率约 50%，和随机猜差不多）
- ❌ 完整的多轮决策链（近一半样本被截断）

## 2. 模型使用场景分析

在讨论如何改进之前，需要先明确**训练出来的模型要用在哪里**，这决定了 playbook 需要达到什么质量标准。

### 2.1 场景 A：客服 Copilot（辅助人工客服）

**用法**：模型作为人工客服的助手，推荐下一步动作，人工客服决定是否采纳。

**对 playbook 质量的要求**：
- 技能映射准确率需要 **≥70%**（推荐的动作大部分要对）
- 动作空间覆盖率需要 **≥90%**（不能经常推荐不出来）
- 对话覆盖率要求不高（Copilot 每轮独立推荐，不需要完整链路）

**当前差距**：技能准确率 47% → 需要提升 23 个百分点；动作空间覆盖 79% → 需要扩充 skill

### 2.2 场景 B：全自动客服 Agent

**用法**：模型完全替代人工，自主完成整个对话。

**对 playbook 质量的要求**：
- 技能映射准确率需要 **≥85%**
- 动作空间覆盖率需要 **≥95%**
- 对话覆盖率需要 **≥80%**（必须能处理长对话）
- 需要真实的自然语言回复能力（当前只选 skill，不生成回复文本）

**当前差距**：全面不达标，且当前框架只选 skill 不生成回复，无法直接用于全自动场景

### 2.3 场景 C：对话策略优化器

**用法**：模型不直接面对买家，而是作为后端策略引擎，决定对话流程的最优路径。前端由规则引擎或另一个 LLM 生成具体回复。

**对 playbook 质量的要求**：
- 技能映射准确率需要 **≥60%**（策略方向对即可）
- 动作空间覆盖率 **≥85%**
- 对话覆盖率需要 **≥70%**

**当前差距**：最接近可用，但技能准确率仍需提升 13 个百分点

### 2.4 推荐路径

**短期（当前轮次）**：瞄准场景 C（策略优化器），这是当前框架最适合的用途。

**中期**：在场景 C 验证有效后，扩展到场景 A（Copilot），需要扩充动作空间和提升映射准确率。

**长期**：场景 B（全自动 Agent）需要根本性的框架改造（从选 skill 变为生成回复），不在本 spec 范围内。

## 3. 质量提升方案

### 3.1 P0：提升技能映射准确率（47% → 70%+）

**根因**：LLM 在将真实 agent 回复映射到 31 个 skill 时，经常映射错误。

**方案 A：改进 LLM 映射 prompt（低成本，预期提升到 55-60%）**

在 `etl/llm_generator.py` 的 playbook 生成 prompt 中：
- 为每个 skill 增加 2-3 个真实客服回复示例（few-shot）
- 增加"反例"说明（容易混淆的 skill 对，如 `gen_clarify` vs `pre_query_product`）
- 要求 LLM 在映射时输出置信度，低置信度的标记为 uncertain

**方案 B：两阶段生成（中等成本，预期提升到 65-70%）**

1. 第一阶段：LLM 生成 playbook（现有流程）
2. 第二阶段：用另一个 LLM 调用做 skill 映射校验
   - 输入：原始 agent 回复 + 分配的 skill
   - 输出：是否正确 + 建议修正

**方案 C：人工校验 + 主动学习（高成本，预期提升到 80%+）**

1. 随机抽样 500 个 playbook，人工校验 skill 映射
2. 用校验结果微调映射 prompt
3. 迭代 2-3 轮

**推荐**：先做方案 A（1-2 天），评估效果后决定是否需要方案 B。

### 3.2 P1：扩充动作空间（覆盖率 79% → 90%+）

**根因**：31 个 skill 无法覆盖真实客服的高频行为。

**审计发现的缺失动作**：

| 真实行为 | 频率 | 建议新增 skill |
|----------|------|---------------|
| 发送商品链接 | 高 | `gen_send_link` |
| 发送图片/视频说明 | 高 | `gen_send_media` |
| 备注订单/特殊处理 | 中 | `gen_note_order` |
| 确认/肯定回复（"是的亲"、"对的"） | 高 | `gen_confirm` |
| 催促/跟进（"帮您催一下"） | 中 | `log_follow_up` |
| 引导加微信/关注店铺 | 中 | `pre_guide_follow` |
| 查询优惠券使用方法 | 低 | 可归入 `pre_check_promo` |

**建议新增 5 个 skill**：`gen_send_link`、`gen_send_media`、`gen_confirm`、`gen_note_order`、`log_follow_up`

**影响范围**：
- `etl/customer_service_env.py` — VALID_SKILLS 集合
- `etl/validator.py` — VALID_SKILLS 集合
- `etl/llm_generator.py` — prompt 中的 skill 列表
- `etl/rl_interfaces.py` — SKILL_ID_TO_IDX 映射
- `etl/prompt_config.py` — skill 描述

**风险**：扩充 skill 后需要重新生成所有 playbook，成本较高。建议先在 100 个样本上验证效果。

### 3.3 P2：减少对话截断（覆盖率 61% → 80%+）

**根因**：`MAX_CONVERSATION_CHARS = 2500` 导致长对话被截断，LLM 只能看到部分内容。

**方案**：
1. 将 `MAX_CONVERSATION_CHARS` 从 2500 提高到 6000
2. 将 `max_tokens`（LLM 输出限制）从 4096 提高到 8192
3. 对超长对话（>8000 字符），改用分段生成策略：
   - 前半段生成前半棵树
   - 后半段生成后半棵树
   - 合并

**预期效果**：路径长度不匹配率从 46.9% 降到 ~20%

**风险**：LLM 调用成本增加，生成时间变长。

### 3.4 P3：过滤低质量样本

**立即可做**：
1. 过滤 `rl_steps <= 1` 的 playbook（632 个，11%）— 对 RL 训练无贡献
2. 过滤 `scenario == "unknown"` 的 playbook（247 个，4.3%）— reward 信号弱
3. 过滤技能映射准确率 < 30% 的 playbook（如果有校验机制）

**预期效果**：训练集从 5732 缩减到 ~4800，但质量显著提升。
